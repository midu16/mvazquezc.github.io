User Namespaces in Kubernetes#

In Kubernetes v1.25 we got a new alpha feature, user namespaces. For those who are not familiar with user namespaces we can briefly describe them as a security mechanism that isolates security-related identifiers and attributes, in particular user IDs and group IDs.

You can read more about user namespaces here.

In the containers space we can leverage user namespaces to run processes as root in the container and with a non-privileged user in the host. The process will have full privileged within the user namespace, but it will be unprivileged within the host.

By using user namespaces we can reduce the damage a compromised container can do to other pods or the host where it runs.

Note

For this post I’ll be running a Kubernetes v1.25 cluster. If you want to try this in your own environment you can use your favorite tool to get a K8s cluster up and running, I’ll be using kcli.

# Create a Kubernetes 1.25 cluster with 1 master and 1 worker using calico as SDN, nginx as ingress controller, metallb for loadbalancer services and CRI-O as container runtime
kcli create kube generic -P masters=1 -P workers=1  -P master_memory=4096 -P numcpus=2 -P worker_memory=4096 -P sdn=calico -P version=1.25 -P ingress=true -P ingress_method=nginx -P metallb=true -P engine=crio -P domain=linuxera.org userns-cluster

User Namespaces support in Container Engines#

One of the requirements to be able to make use of user namespaces is having support for them in the container engine that our cluster uses.

At the time of this post, support for user namespaces only made it to CRI-O. ContainerD user namespaces support is planned for the v1.7 release which is not released yet.

In case you create a pod requesting user namespaces on a cluster running a container engine without support for them, the pod will be created without user namespaces support.

Limitations#

As we can read in the official docs there are some limitations when a workload leverages user namespaces.

Pod will not be able to set any of:

  • hostNetwork: true
  • hostIPC: true
  • hostPID: true

No stateful volumes can be used, only the following volumes are allowed:

  • configmap
  • secret
  • projected
  • downwardAPI
  • emptyDir

User Namespaces in Action#

We will deploy the same workload using user namespaces and not using user namespaces, then we will check the main differences at host level. For the workload we will be using a simple go app that exposes a service on a port of our choice.

Before deploying the workloads, make sure you have enabled the UserNamespacesStatelessPodsSupport feature gate in the Kubernetes API Server. This feature is currently in alpha and disabled by default.

You can enable the feature gate by adding the line below to the API Server. Remember that this feature is NOT ready for production usage.

--feature-gates=UserNamespacesStatelessPodsSupport=true

Workload using user namespaces#

In this example we’re going to deploy our workload in a namespace that enforces the privileged standard and audits/warns the restricted standard.

  1. Create the namespace for our workloads:

    cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: user-namespaces-tests
EOF

  2. Create the workload:

    Note

    User namespaces are enabled by setting the .spec.hostUsers field of the pod to false.

    cat <<EOF | kubectl -n user-namespaces-tests apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: go-app
  name: go-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: go-app
  strategy: {}
  template:
    metadata:
      labels:
        app: go-app
    spec:
      hostUsers: false
      containers:
      - image: quay.io/mavazque/reversewords:ubi8
        name: reversewords
        resources: {}
        securityContext:
          runAsUser: 0
EOF

  3. Now that the workload is running, let’s exec into the pod and

kubectl -n user-namespaces-tests exec deploy/go-app -ti – /bin/bash

id uid=0(root) gid=0(root) groups=0(root)
